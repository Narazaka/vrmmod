# MC 1.20.1 Support Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add MC 1.20.1 (Fabric + Forge) support to the existing Stonecutter multi-version build.

**Architecture:** Extend the existing Stonecutter conditional compilation pattern established in Phase 2 (1.21.1). Add new version constants for 1.20.1-specific API differences. The majority of changes are small (ResourceLocation constructor, VertexConsumer method names, BlockPos factory). The largest single change is the networking API (Architectury v9 uses a different registration pattern). GUI code requires no changes (GuiGraphics exists in 1.20.1).

**Tech Stack:** Stonecutter 0.8.3, Stonecraft 1.9.x, Architectury API v9.2.14 (for 1.20.1), Forge 47.3.0, Java 17

---

## Scope Assessment

**当初の想定 vs 実際:**

| 項目 | 当初の予想 | 実際の調査結果 |
|------|-----------|---------------|
| GUI (VrmModScreen, VrmSettingsList) | 全面書き換え | **変更不要** (GuiGraphics は 1.20 で導入済み) |
| Player API (walkAnimation等) | 大幅変更 | **変更不要** (1.20.1 に存在) |
| ネットワーク | 全面書き換え | **書き換え必要** (CustomPacketPayload → 旧 Architectury API) |
| Forge エントリポイント | 新規作成 | 新規作成（NeoForge と似た構造） |
| VertexConsumer | 未調査 | **メソッド名変更** (addVertex→vertex, endVertex追加等) |
| ResourceLocation | ファクトリ変更 | `fromNamespaceAndPath` → コンストラクタ (4箇所) |
| BlockPos | ファクトリ変更 | `BlockPos.containing` → `BlockPos` コンストラクタ (2箇所) |

**工数評価: Medium（当初 Very High → 大幅に下方修正）**

## API 差分一覧

### 1. ResourceLocation ファクトリ (1.21+ vs 1.20.1)

```kotlin
// 1.21+
ResourceLocation.fromNamespaceAndPath("vrmmod", "model_announce")
ResourceLocation.withDefaultNamespace("textures/misc/unknown_pack.png")

// 1.20.1
ResourceLocation("vrmmod", "model_announce")
ResourceLocation("textures/misc/unknown_pack.png")
```

**影響ファイル:** ModelAnnouncePayload.kt, PlayerModelPayload.kt, VrmTextureManager.kt, VrmRenderer.kt

### 2. VertexConsumer API (1.21+ vs 1.20.1)

```kotlin
// 1.21+
vertexConsumer
    .addVertex(pose, px, py, pz)
    .setColor(r, g, b, 255)
    .setUv(u, vCoord)
    .setOverlay(OverlayTexture.NO_OVERLAY)
    .setLight(packedLight)
    .setNormal(pose, nx, ny, nz)

// 1.20.1
vertexConsumer
    .vertex(pose.pose(), px, py, pz)
    .color(r, g, b, 255)
    .uv(u, vCoord)
    .overlayCoords(OverlayTexture.NO_OVERLAY)
    .uv2(packedLight)
    .normal(pose.normal(), nx, ny, nz)
    .endVertex()
```

**注意:** 1.21+ の `setNormal(PoseStack.Pose, ...)` は `pose` オブジェクトを直接渡す。1.20.1 の `normal(Matrix3f, ...)` は `pose.normal()` で Matrix3f を渡す。同様に `addVertex(PoseStack.Pose, ...)` vs `vertex(Matrix4f, ...)`。

**影響ファイル:** VrmRenderer.kt (drawPrimitive メソッド、1箇所)

### 3. ネットワーク API (Architectury v13+ vs v9)

```kotlin
// 1.21+ (Architectury v13+): CustomPacketPayload + StreamCodec
data class ModelAnnouncePayload(...) : CustomPacketPayload {
    companion object {
        val TYPE = CustomPacketPayload.Type<ModelAnnouncePayload>(ResourceLocation.fromNamespaceAndPath(...))
        val CODEC: StreamCodec<RegistryFriendlyByteBuf, ModelAnnouncePayload> = ...
    }
}
NetworkManager.registerReceiver(NetworkManager.c2s(), TYPE, CODEC) { payload, context -> ... }
NetworkManager.sendToServer(payload)

// 1.20.1 (Architectury v9): ResourceLocation + FriendlyByteBuf
val PACKET_ID = ResourceLocation("vrmmod", "model_announce")
NetworkManager.registerReceiver(NetworkManager.Side.C2S, PACKET_ID, CODEC) { buf, context ->
    val payload = decode(buf)
    context.queue { handle(payload) }
}
NetworkManager.sendToServer(PACKET_ID) { buf -> encode(buf, payload) }
```

**影響ファイル:** ModelAnnouncePayload.kt, PlayerModelPayload.kt, VrmModNetwork.kt, VrmModServer.kt, VrmModClient.kt

### 4. BlockPos.containing (1.20.2+ vs 1.20.1)

```kotlin
// 1.20.2+
BlockPos.containing(player.x, player.eyeY, player.z)
BlockPos.containing(loc)  // Vec3

// 1.20.1
BlockPos(player.x.toInt(), player.eyeY.toInt(), player.z.toInt())
BlockPos(loc.x.toInt(), loc.y.toInt(), loc.z.toInt())
```

**注意:** `BlockPos.containing` は floor を使う。1.20.1 では `Mth.floor()` を使うか、直接 `BlockPos(Mth.floor(x), Mth.floor(y), Mth.floor(z))` とする。

**影響ファイル:** VrmFirstPersonRenderer.kt, MixinHelper.kt

### 5. Fabric WorldRenderContext tickDelta (1.21+ vs 1.20.x)

```kotlin
// 1.21+
context.tickCounter().getGameTimeDeltaPartialTick(false)

// 1.20.x
context.tickDelta()
```

**影響ファイル:** VrmModFabric.kt

### 6. Forge エントリポイント (1.20.1 のみ)

NeoForge (`net.neoforged.*`) の代わりに Forge (`net.minecraftforge.*`) を使用。

**新規ファイル:** VrmModForge.kt, META-INF/mods.toml

### 7. GameRendererMixin: pickRange (既存 HAS_INTERACTION_RANGE で対応済み?)

```java
// 1.20.5+ (HAS_INTERACTION_RANGE)
double blockRange = minecraft.player.blockInteractionRange();
double entityRange = minecraft.player.entityInteractionRange();

// 1.20.1
double blockRange = minecraft.gameMode.getPickRange();
double entityRange = blockRange;  // 1.20.1 doesn't separate block/entity range
```

**影響ファイル:** GameRendererMixin.java

### 8. Mixin JSON: Java 互換レベル

```json
// 1.21+
"compatibilityLevel": "JAVA_21"

// 1.20.1
"compatibilityLevel": "JAVA_17"
```

**影響ファイル:** vrmmod.mixins.json

### Task 9: Mixin JSON 互換レベル

**Files:**
- Modify: `src/main/resources/vrmmod.mixins.json`

Stonecutter は JSON ファイルもプリプロセスできる（`json5` ハンドラ経由）。ただし、`vrmmod.mixins.json` は標準 JSON なので Stonecutter コメントは使えない。

**代替案:** `build.gradle.kts` の `processResources` で `compatibilityLevel` を置換する。

```kotlin
tasks.processResources {
    filesMatching("vrmmod.mixins.json") {
        val javaVersion = if (stonecutter.eval(stonecutter.current.version, ">=1.20.6")) "JAVA_21" else "JAVA_17"
        filter { it.replace("JAVA_21", javaVersion) }
    }
}
```

**注意:** Stonecraft が既に processResources をカスタマイズしている可能性がある。衝突しないよう `afterEvaluate` 内で追加するか、確認すること。
